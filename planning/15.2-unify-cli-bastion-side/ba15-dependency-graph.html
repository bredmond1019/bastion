Will mev validate-brain, manifest, emit-graph and emit-state still work after the dedup?

Yes — the standalone mev binary and its four commands keep their exact CLI surface and output. BA.15.12 only swaps what's behind them: mev's own duplicate OKF/state code gets deleted and repointed at okf-core as a library. Nothing about how you call mev today changes.

Answer

The mev binary is never removed or renamed. BA.15.12's own acceptance criteria require bastion validate-brain to stay byte-identical to mev validate-brain on the whole brain corpus — so mev's output is the parity bar the work has to clear, not something that's allowed to drift. What moves is the ~6,300 lines of duplicate parser/schema code living in mev/src/brain/okf.rs and mev/src/brain/state.rs — that gets deleted and replaced with calls into okf-core, which mev will depend on as a new Cargo path dependency. Same commands in, same diagnostics out.

Why the mev CLI is safe to keep depending on

  • Behavior is the acceptance bar, not a hoped-for side effect. D15 and the master-plan's BA.15.12 entry both state the AC explicitly: mev's OKF/state dupes deleted, bastion validate-brain output parity with mev on the whole brain corpus, combined test count not lower, gated checks green in both repos.
  • It's deferred on purpose. D15 calls this "the risky half" precisely because mev's graph + state validation is proven and load-bearing — the plan will not touch it until there's real appetite and a scoped task spec, not opportunistically.
  • No workspace-absorb. mev stays its own repo with its own binary — BA.15.12 only adds one new path dependency (okf-core) to it; it does not fold mev into the bastion workspace.
